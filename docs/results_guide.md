# Results Guide

> How to interpret output files, metrics, and plots generated by training.

---

## 1. Output Files

After training or evaluation, the `results/` directory will contain the following files (depending on what you ran):

| File | Created by | Description |
|------|-----------|-------------|
| `q_table.npy` | `q_learning_agent.py` | Saved Q-table as NumPy array of shape `(num_states, num_actions)` |
| `q_learning_log.csv` | `q_learning_agent.py` | CSV file with two columns: `step` and `avg_reward` (rolling average) |
| `dqn_model.keras` | `deep_q_agent.py` | Saved Keras model file |
| `dqn_log.csv` | `deep_q_agent.py` | CSV file with two columns: `step` and `avg_reward` (rolling average) |
| `convergence_plot.png` | `train.py --plot` | Plot showing reward vs. training steps for one or both agents |
| `policy_heatmap.png` | `evaluate.py --heatmap` | Visualization of greedy policy for all states |

---

## 2. CSV Log Format

Both `q_learning_log.csv` and `dqn_log.csv` have the same simple two-column format:

```csv
step,avg_reward
10000,0.8234
20000,1.2451
30000,1.5678
...
1000000,2.3871
```

- **`step`**: The training step number at which the log entry was recorded (every `log_interval` steps)
- **`avg_reward`**: The rolling average reward over the last `eval_window` steps (window size is 1000 by default)

### Loading Logs in Python

You can load and plot these logs easily using `pandas` and `matplotlib`:

```python
import pandas as pd
import matplotlib.pyplot as plt

# Load Q-Learning log
df_q = pd.read_csv("results/q_learning_log.csv")

# Plot
plt.figure(figsize=(10, 6))
plt.plot(df_q["step"], df_q["avg_reward"], label="Q-Learning")
plt.xlabel("Training Step")
plt.ylabel("Average Reward (packets/step)")
plt.title("Q-Learning Convergence")
plt.legend()
plt.grid(True, alpha=0.3)
plt.show()
```

---

## 3. Interpreting the Convergence Plot

The convergence plot shows how the agent's average reward improves over time. A typical plot has three phases:

1. **Early Exploration Phase**: ε is high, agent acts mostly randomly → low average reward
2. **Learning Phase**: Q-values (or network weights) start to converge → reward increases steadily
3. **Converged Phase**: ε is near minimum, agent mostly exploits → reward plateaus near optimal value

### What Is a Good Final Reward?

| Average Reward | Interpretation |
|----------------|----------------|
| < 1.0 | Agent is not learning well. Check hyperparameters or train longer. |
| 1.5 - 2.5 | Typical converged performance. Agent is using a good mix of actions. |
| > 2.5 | Excellent performance. Agent is effectively using both active transmit and backscatter. |

## 4. Interpreting the Policy Heatmap

The policy heatmap shows the greedy optimal action for every state `(j, d, e)`, organized into two separate plots for jammer idle (`j=0`) and jammer active (`j=1`). Each cell in the heatmap corresponds to a particular data queue level `d` (rows) and energy queue level `e` (columns). The color of the cell indicates which action is best in that state.

### Expected Patterns
For jammer idle (`j=0`):
- If `d>0` and `e ≥ e_t`: Action 1 (Active Transmit) should dominate
- Else: Action 0 (Idle) is common

For jammer active (`j=1`):
- If `d>0` and power is low: Action 3 (Backscatter)
- If `e` is low: Action 2 (Harvest Energy)
- Other cases: Actions 4-6 (Rate Adaptation)

## 5. Action Distribution (Evaluation Output)

When you evaluate an agent (either via `train.py --eval` or `evaluate.py`), the output will include a breakdown of how often each action was used:

```
Evaluation (50000 steps)
  Average throughput : 2.3712 packets/step
  Action distribution:
    [0] Idle          :   3214  ( 6.4%)
    [1] Active-TX     :  18432  (36.9%)
    [2] Harvest       :   4101  ( 8.2%)
    [3] Backscatter   :  15623  (31.2%)
    [4] RA-0          :   5812  (11.6%)
    [5] RA-1          :   2081  ( 4.2%)
    [6] RA-2          :    737  ( 1.5%)
```

- High percentage for **Active-TX** and **Backscatter**: Agent is aggressive about sending packets
- **Harvest** usage: Agent is managing energy reserves for when jammer is idle
- **Rate Adaptation 2** should be rare because it delivers zero packets

## 6. Metrics Dictionary

When you run training, the `compute_metrics` function calculates several useful metrics:

| Metric Key | Description |
|------------|-------------|
| `total_steps` | Total number of training steps |
| `mean_reward` | Mean reward over all training steps |
| `std_reward` | Standard deviation of rewards (measures variability) |
| `max_reward` | Highest reward achieved in a single step |
| `final_avg` | Mean reward over the last `eval_window` steps (best measure of final performance) |
| `zero_reward_pct` | Percentage of steps where reward was zero (indicates how often agent is idle or fails) |

## 7. Troubleshooting

If your agent isn't learning well, check these common issues:

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| Reward stays near zero | Exploration rate (ε) not decaying or decaying too slowly | Check `epsilon_decay` and `epsilon_end` in `parameters.py` |
| Reward oscillates and never plateaus | Learning rate is too high | Reduce `learning_rate_Q` (for Q-Learning) or `learning_rate_DQN` (for DQN) |
| DQN doesn't converge or diverges | Target network not being updated often enough | Reduce `target_update_freq` in `parameters.py` |
| Training takes too long | Number of training steps set too high | Reduce `T` (Q-Learning) or `T_DQN` (DQN) for quick tests |
| ImportError: No module named 'tensorflow' | TensorFlow not installed | Install TensorFlow: `pip install tensorflow==2.9.1` |
| Policy heatmap looks random | Q-table not trained long enough | Train agent for more steps, or check learning rate |

## 8. Extending the Experiments

Here are some ways you can extend the experiments and explore further:

### Vary Jammer Aggressiveness
Edit `nu` in `parameters.py`:
- `nu = 0.5`: Jammer is idle 50% of the time (easier problem)
- `nu = 0.01`: Jammer is almost always active (harder problem)

### Vary Queue Sizes
Edit `d_queue_size` and `e_queue_size` in `parameters.py`:
- Larger queues let agent buffer more packets and energy
- Note: Changing queue sizes changes `num_states`, which invalidates saved models!

### Change Packet Arrival Rate
Edit `arrival_rate` in `parameters.py`:
- Lower rate (`1-2`): Lighter traffic, queues rarely fill up
- Higher rate (`5-8`): Heavier traffic, queues often full

### Compare Multiple Seeds
To get statistically meaningful results, run multiple training runs with different seeds:

```bash
# Run 5 different seeds for Q-Learning
for seed in 0 1 2 3 4; do
    python train.py --agent q --seed $seed
    mv results/q_learning_log.csv results/q_log_seed$seed.csv
    mv results/q_table.npy results/q_table_seed$seed.npy
done
```

Then you can load all logs, compute mean and standard deviation, and plot with error bars for confidence intervals.
